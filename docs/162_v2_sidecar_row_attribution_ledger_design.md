# V2 Sidecar Row Attribution Ledger Design

Date: 2026-06-19 Asia/Seoul

Status: `design_only_pre_implementation`

## Purpose

This document defines the design-only schema and test plan for the V2 sidecar
`row_attribution_ledger` approved in
`docs/161_v2_open_decisions_review.md`.

The business goal is to let a reviewer answer which operation affected a row
without exposing raw exact keys or weakening the existing
`all_metrics(timestamp, device_id)` upsert safety mechanism.

This document does not approve code changes, database migrations, Supabase
schema changes, operational DB access, fixture DB mutation, destructive smoke
tests, LAN exposure, delete UI changes, release packaging, commit, push, or PR
creation.

## Ledger Model

The ledger is a sidecar table in the local state DB. It records attribution
evidence for upload, delete, and reconcile outcomes without storing raw business
keys.

The ledger is append-only. Corrections, recovery, and reconcile results are
represented by later records that point to the same operation and safe key hash.
Existing ledger rows are not updated or deleted as part of normal operation.

Logical table:

```text
row_attribution_ledger(
  attribution_id,
  operation_id,
  operation_type,
  operation_phase,
  audit_id,
  db_delta_id,
  actor_id,
  actor_role,
  exact_key_hash,
  exact_key_hash_version,
  source_evidence_hash,
  source_evidence_hash_version,
  outcome,
  reason_code,
  db_target_class,
  db_fingerprint_hash,
  schema_fingerprint_hash,
  supersedes_attribution_id,
  created_at
)
```

This is a logical schema, not executable DDL.

## Field Definitions

| Field | Required | Design meaning |
| --- | --- | --- |
| `attribution_id` | yes | Stable local ledger id, generated by the local state DB layer. |
| `operation_id` | yes | Upload job id, delete run id, reconcile id, or future operation id that caused the attribution evidence. |
| `operation_type` | yes | Safe operation class such as `upload_start`, `upload_retry`, `delete_start`, or `delete_reconcile`. |
| `operation_phase` | yes | Phase that produced the evidence: `before_mutation`, `after_mutation`, `reconcile`, or `blocked`. |
| `audit_id` | yes | Append-only audit row that records the same operation decision or result. |
| `db_delta_id` | no | DB delta evidence id when a before/after DB delta measurement exists. |
| `actor_id` | yes | Safe actor identifier from the current local context. It must not be a raw OS username unless that value is separately approved for audit storage. |
| `actor_role` | yes | Safe role label captured as evidence. This field does not approve a final LAN role matrix. |
| `exact_key_hash` | yes | HMAC-derived safe hash of canonical `(timestamp, device_id)`. |
| `exact_key_hash_version` | yes | Hash algorithm and canonicalization version, for example `hmac-sha256-v1`. |
| `source_evidence_hash` | yes | Safe hash of normalized source evidence. It must not reveal raw source paths, operational filenames, or CSV row contents. |
| `source_evidence_hash_version` | yes | Source evidence canonicalization and hash version. |
| `outcome` | yes | One allowed ledger outcome from the outcome table below. |
| `reason_code` | no | Safe reason code for blocked, failed, unknown, or reconciled outcomes. |
| `db_target_class` | yes | Safe DB target class such as local operational, local fixture, or blocked target class. |
| `db_fingerprint_hash` | yes | Safe hash proving the DB target class/fingerprint used by the operation. |
| `schema_fingerprint_hash` | yes | Safe hash proving the expected DB schema fingerprint, without storing raw SQL. |
| `supersedes_attribution_id` | no | Earlier local ledger id superseded by a later recovery/reconcile evidence row. |
| `created_at` | yes | UTC timestamp when the local ledger row was written. |

## Allowed Outcomes

| Outcome | Meaning |
| --- | --- |
| `inserted` | The operation inserted a previously absent DB row. |
| `upsert_accepted` | The upload path accepted the row for upsert, but the DB row may have been inserted or updated by existing DB semantics. |
| `unchanged` | The operation found an existing row and made no effective row change. |
| `deleted` | The approved destructive operation deleted the row. |
| `reconciled_absent` | Read-only reconcile evidence proved the row is absent after an unknown or failed path. |
| `blocked` | The operation was blocked before mutation. |
| `failed_before_mutation` | The operation failed before any DB mutation could start. |
| `unknown_requires_reconcile` | The operation may have committed or partially finalized, so read-only reconcile is required. |

The ledger must not infer success from missing evidence. Unknown states stay
unknown until a read-only reconcile path records safe evidence.

## Safe Hash Design

`exact_key_hash` is derived from the canonical `(timestamp, device_id)` pair
using an installation-local HMAC key. The stored value is the digest and version
only.

Canonical key input rules:

- timestamp uses the same normalized value used by the upload/delete exact-key
  path;
- device id uses the same normalized value used by the upload/delete exact-key
  path;
- the delimiter and field order are versioned by `exact_key_hash_version`;
- the raw timestamp and raw device id are never stored in the ledger, API
  responses, audit rows, logs, screenshots, generated reports, or test output.

`source_evidence_hash` is derived from normalized source evidence that can be
recomputed without storing raw source paths or operational filenames. The
canonical source evidence bundle may include safe identifiers and digests such
as operation id, preview run id, source item id, parser version, normalized row
count, content digest, and selected date-scope metadata when separately
approved.

Forbidden ledger fields and outputs:

- raw `(timestamp, device_id)` values;
- raw exact keys;
- raw source paths;
- operational filenames;
- CSV row content;
- raw SQL;
- DB URLs;
- tokens;
- Authorization values;
- JWTs;
- anon keys;
- service role values;
- secrets;
- raw exception payloads.

## Audit And DB Delta Links

Each attribution row must link to an audit row through `audit_id`.

For operations with measurable before/after database evidence, each attribution
row should also link to `db_delta_id`. If DB delta cannot be measured, the
operation must record the safe reason in DB delta evidence and the attribution
row must use a safe `reason_code`.

Write order constraints:

- destructive mutation must not start unless the required audit-start row has
  been written;
- attribution must not report `deleted`, `inserted`, `upsert_accepted`, or
  `unchanged` without an operation id and audit id;
- attribution must not report success when DB delta evidence contradicts the
  expected operation result;
- if commit state is unknown, attribution uses `unknown_requires_reconcile`
  until read-only reconcile writes later evidence.

## Recovery And Reconcile Behavior

Recovery and reconcile do not mutate Supabase rows for attribution. They write
local state evidence only.

Reconcile mapping:

| Operation state | Ledger behavior |
| --- | --- |
| `commit_unknown` | Write `unknown_requires_reconcile` for affected safe key hashes when safe evidence exists. |
| `reconciling` | Do not rewrite prior outcomes. Write progress evidence only if a safe audit row exists. |
| `reconciled_succeeded` | Write later `deleted`, `inserted`, `unchanged`, or `reconciled_absent` rows as supported by read-only evidence. |
| `reconciled_rolled_back` | Write `unchanged` or equivalent safe outcome rows supported by read-only evidence. |
| `reconciliation_failed` | Keep previous unknown evidence and write safe failure reason codes. |

The ledger must preserve enough safe evidence to explain why future destructive
operations remain blocked after unresolved unknown or reconciliation-failed
states.

## Review Queries

Future read paths may answer these questions from the local state DB without
exposing raw keys:

- which operation id affected this safe key hash;
- which audit id recorded the operation;
- which actor id and role were recorded for the operation;
- which DB delta evidence id measured the row-count change;
- whether the latest safe outcome is inserted, upsert-accepted, unchanged,
  deleted, reconciled absent, blocked, failed before mutation, or unknown.

No operator-facing query may reverse a safe hash into raw `(timestamp,
device_id)` values.

## Test Plan

This plan defines future tests only. It does not execute tests, mutate DBs, or
approve implementation.

Required non-destructive tests:

1. Schema contract tests prove the logical fields above are represented in the
   local state DB implementation and no Supabase table is changed.
2. Hash determinism tests prove the same canonical synthetic exact key produces
   the same `exact_key_hash` for one installation and hash version.
3. Hash separation tests prove different canonical synthetic exact keys produce
   different safe hashes.
4. Redaction tests prove raw exact keys, raw source paths, operational
   filenames, CSV row content, raw SQL, DB URLs, tokens, Authorization values,
   JWTs, anon keys, service role values, secrets, and raw exception payloads do
   not appear in ledger rows, API responses, audit rows, logs, screenshots, or
   generated reports.
5. Audit link tests prove every ledger row has an `audit_id` and that audit
   write failure before destructive mutation blocks the mutation path.
6. DB delta link tests prove measurable operations link attribution rows to
   `db_delta_id` and mismatched delta evidence prevents success attribution.
7. Append-only tests prove normal correction, recovery, and reconcile paths
   create later rows instead of updating or deleting existing attribution rows.
8. Recovery tests prove `commit_unknown` writes
   `unknown_requires_reconcile` and blocks later destructive actions until
   read-only reconcile records a safe outcome.
9. Reconcile tests prove source/keyset rebuild failure, DB target drift, schema
   drift, or DB count failure records safe failure reason codes and does not
   infer success.
10. Compatibility tests prove the design does not weaken
    `all_metrics(timestamp, device_id)` uniqueness, upsert, or de-dup behavior.
11. Marker-scan tests run against diffs and generated artifacts to detect unsafe
    sensitive markers before merge.

## Acceptance Criteria Before Implementation

Future implementation may start only after a separate approval confirms:

- this logical schema is accepted or explicitly revised;
- any local state DB migration plan is reviewed separately;
- no Supabase schema migration is included;
- audit and DB delta record ids exist before success attribution is written;
- safe hash versions and canonicalization rules are testable;
- recovery and reconcile behavior is covered by non-destructive tests;
- marker scans cover source diffs, generated reports, logs, and screenshots;
- rollback for any local state DB change is defined before merge.
